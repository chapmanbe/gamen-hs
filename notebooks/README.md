# gamen-hs notebooks

Pedagogical chapters in plain Markdown with executable Haskell code
blocks. Authored to be read alongside an open GHCi session against
the `gamen` library; every code block tagged `{.eval}` has its
expected output captured in the rendered HTML so the reader can
compare what their shell produces.

## Authoring conventions

| Block tag | Treatment |
|---|---|
| ` ```haskell ` | Definition or import. Kept in the program context for later `{.eval}` blocks; rendered for the reader. |
| ` ```haskell {.eval} ` | Single expression. The build script wraps it in `print (...)` and runs it; the captured output is spliced in as a sibling ` ```output ` block. |
| ` ```haskell {.ghci} ` | GHCi directive (`:set`, `:t`, `:l`). Rendered for the reader but neither accumulated nor evaluated. |
| ` ```bash ` | Shell command. Rendered as-is. |
| ` ```output ` | Generated by the build script — do not edit by hand. |

Sidenotes and margin notes follow chapmanbe.github.io's convention:

```liquid
{% include sidenote.html id="some-id" content="Numbered aside in the margin." %}
{% include marginnote.html id="other-id" content="Unnumbered aside (figure caption etc.)." %}
```

## Local build & view

The build script is idempotent — it strips any previously-spliced
`output` blocks before re-running, so the chapter source on disk is
always the latest rendered version.

```bash
make -f notebooks/Makefile build    # rewrites notebooks/01-kripke.md in place
make -f notebooks/Makefile serve    # build + jekyll serve at :4000
make -f notebooks/Makefile clean    # remove build/ and _site/
```

The `serve` target runs `jekyll serve` and opens a watch loop. Edit
the Markdown, save, and Jekyll auto-reloads — but note that Jekyll
won't re-run `build.py`, so if you change a `{.eval}` block you need
to rerun `make build` to refresh the captured outputs.

### Ruby setup

The system Ruby on macOS (2.6.x) is too old. Use Homebrew's:

```bash
brew install ruby
export PATH="$(brew --prefix ruby)/bin:$(brew --prefix)/lib/ruby/gems/4.0.0/bin:$PATH"
make -f notebooks/Makefile install   # one-time bundle install
```

The Makefile detects the Homebrew Ruby paths automatically; you only
need the `PATH` export in your shell rc if you want to run
`bundle`/`jekyll` directly.

### Sidecar `.hs`

Each build produces `notebooks/build/<chapter>.hs` (gitignored) with
every non-eval code block concatenated. Readers who'd rather load
the chapter than copy-paste it can:

```bash
cabal exec -- runghc --ghc-arg=-package --ghc-arg=gamen notebooks/build/01-kripke.hs
```

CI can use the same sidecar as a compile-check.

## Deployment to GitHub Pages

The Pages workflow in `.github/workflows/docs.yml` currently only
builds Haddock and ships it to the project root
(`chapmanbe.github.io/gamen-hs/`). To also publish the notebooks
under `/notebooks/`:

1. Add a build step that runs `notebooks/build.py` over each
   chapter, producing `notebooks/build/*.md`.
2. Add a Jekyll build step (e.g. via `actions/jekyll-build-pages@v1`)
   pointed at `notebooks/` as the source directory and
   `notebooks/_site/` as the destination.
3. Copy `notebooks/_site/*` into the deployment artifact under
   `notebooks/`, leaving the Haddock output at the root.
4. Verify that Jekyll's processing does not touch the Haddock
   directory by listing it in `_config.yml`'s `exclude:`.

## Why no Pandoc?

`chapmanbe.github.io` is already a Jekyll site, and the gamen-hs
Pages output is intended to render against the same Tufte styling.
Putting Markdown into Jekyll is the shortest path; Pandoc would be
an extra moving part. The trade-off: Jekyll's Liquid templating is
less expressive than Pandoc filters, but for the modal-logic content
it does fine — sidenotes, code highlighting via Rouge, and
straightforward layouts cover everything we need.

## Why not literate Haskell?

The reader's workflow is a GHCi shell with copy-paste, not
`ghc chapter.lhs`. Plain Markdown keeps narrative clean and Pandoc-
or Jekyll-rendering free of `.lhs`-specific quirks. The build
script's sidecar `.hs` file gives back the "compile-checks the whole
chapter" property without the lhs source-file convention.
